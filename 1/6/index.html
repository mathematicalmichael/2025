Analyzing Michel Gondry's Work

2025-01-06

Continuing the work from yesterday, today’s test video has a lot of very artistic stuff going on despite the fixed camera positions:

  • both static and dynamic scenes full of motion
  • rotations of the camera
  • close-ups and long shots
  • multiple people in the scene
  • it’s unclear what is happening in any given frame; temporal context is important

I’ll be connecting the posts from the Kosmos-2 API and Text Similarity posts (using the same patterns), but on the data of a new music video.

To recap - I’m splitting up the video into its component frames, and then using the Kosmos-2 API to get a description of each frame. I then render a new frame using the annotations from the model, and stitch the frames together to form a new video (the one you see above).

Michel Gondry videos have an entrancing quality to them, and I hope the annotations don’t take away from that. I present to you some curated highlights (both good and bad), without commentary (I’ll let the model provide that with the captions):

Okay… some commentary: This seems to be bouncing around from “almost perfect” to “hilariously off.” Just like the White Stripes video from 1/5, there is a lot of variance even when the frames are only subtly different.

This got me thinking about how I could actually visualize the captions provided by the model by connecting it to the embedding work from 1/3.

I dove down the rabbit hole of visualizing this video as a single image, as a prototype of the general pipeline I’m working on.

Right now the primary objective is to get a “feel” for whether the general framework and architecture is making some sense.

I’ll know that things are going well (without defining a formal metric), if I see that my model can embed the Kosmos-2 descriptions such that examples from the training data appear similar to the frame descriptions. This model will be trained on a large collection of human-annotated image captions.

Technical Notes

I decided to sort of stray away from software-engineering “best practices” on this one. Instead, I chose to optimize for absolute brevity (plotting code aside). What I mean by this statement is that instead of writing importable and re-usable functions, I instead wrote the shortest possible python scripts that would do the job.

Some choices I made for this exercise:

  • Use sys.argv to pass arguments to the script in favor of argparse or click.
  • Don’t write functions that I maybe will import elsewhere, just the bare minimum.
  • Try to prepare data with CLI tools.
  • Documentation should basically be cat <file>. If you can’t understand it at a glance, it’s doing too much.
  • Take inspiration from modularity - single responsibility per file; minimal arguments; minimal inputs (this will play nicely with the makefile operations).

I managed to keep things to “under 42 lines of code” (excluding plotting), with the general end-to-end flow looking like this:

$ cat reduce.py transform.py txt2vec.py | wc -l
34

$ cat 42.sh
seq -w 1 9 | xargs -I {} curl -s https://storage.googleapis.com/localized-narratives/annotations/open_images_train_v6_localized_narratives-0000{}-of-00010.jsonl | jq .caption | sed 's|"||g' >> train.txt
curl -fsSL https://cdn.math.computer/v/kosmos2/go/frames.tar.gz | tar xvz - | sort | tail -n+2 | xargs -I {} jq .output {} | sed 's|"||g' | awk -F "Describe the scene in detail: " '{ print $2 }' > data.txt
uv run --extra embed txt2vec.py train.txt
uv run --extra embed txt2vec.py data.txt
uv run --extra embed reduce.py train
uv run --extra embed transform.py train data
uv run --extra embed vec2html.py dataXtrain 950  # full screen
uv run --extra embed vec2html.py dataXtrain 380  # preview
  1. Download the training captions using curl, jq, and sed
  2. Download the extracted still frames using curl, tar, sort, tail, jq, sed, and awk
  3. Embed the training captions using txt2vec.py
  4. Embed the extracted still frames using txt2vec.py
  5. Reduce the training captions using reduce.py
  6. Transform the training captions using transform.py
  7. Visualize the Kosmos-2 frame descriptions using vec2html.py

(with the last step being generalizable to “visualizing the training data” as well as “visualizing the frame descriptions when training the model on solely them”).

Looking at the result, there are two areas where most samples lie. The denser of the two (just barely), is the one on the right side:

It’s interesting to see the separation between the two regions; it’s hard to label “why.” Qualitatively, the one on the right seemed to have more “good” captions, and the symmetry of the cluster was curious. Additionally, many samples on the outskirts happen to correspond to poor quality captions.

As soon as I hovered over this region (I encourage you to do the same by clicking the button above to the full visualization), I started noticing captions that looked to be on the right track.

It took less than a minute or two to find all of these screen-captures.

These are some great initial results! I’m really pleased, as this seem fairly coherent. What I’m neglecting to show, however, is that many of the video frames got embedded with really poor captions.

It’s an interesting topic to consider how the embedding visualization can help filter out “good data” from “bad data” in some automated way.

For the curious, here is the makefile that generated the above results, which is responsible for avoiding re-running the same operations on the same data unless something relevant has changed:

train.txt:
	seq -w 1 9 | xargs -I {} curl -s https://storage.googleapis.com/localized-narratives/annotations/open_images_train_v6_localized_narratives-0000{}-of-00010.jsonl | jq .caption | sed 's|"||g' >> train.txt

data.txt:
	curl -fsSL https://cdn.math.computer/v/kosmos2/go/frames.tar.gz | tar xvz - | sort | tail -n+2 | xargs -I {} jq .output {} | sed 's|"||g' | awk -F "Describe the scene in detail: " '{ print $2 }' > data.txt

train.npy: train.txt txt2vec.py
	uv run --extra embed txt2vec.py train.txt

data.npy: data.txt txt2vec.py
	uv run --extra embed txt2vec.py data.txt

# train on caption dataset
train.pkl train.ann: train.npy reduce.py
	uv run --extra embed reduce.py train

dataXtrain.npy: transform.py data.npy train.pkl
	uv run --extra embed transform.py train data

dataXtrain_950.html: vec2html.py dataXtrain.npy
	uv run --extra embed vec2html.py dataXtrain 950

dataXtrain_380.html: vec2html.py dataXtrain.npy
	uv run --extra embed vec2html.py dataXtrain 380

# train on frames dataset
data.pkl data.ann: data.npy reduce.py
	uv run --extra embed reduce.py data

dataXdata.npy: transform.py data.npy data.pkl
	uv run --extra embed transform.py data data

dataXdata_950.html: vec2html.py dataXdata.npy
	uv run --extra embed vec2html.py dataXdata 950

dataXdata_380.html: vec2html.py dataXdata.npy
	uv run --extra embed vec2html.py dataXdata 380
