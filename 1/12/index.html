Maturing the Video-to-Image Pipeline

2025-01-12

The problem I am starting to encounter (as I have many times before), is that after only a few days of artistic experiments (especially in the early prototyping stages), my working directory begins to look like chaos.

Here are just temporary files created in order to share images on this site:

$ ls *.html *.pkl *.ann *npy *.svg *.png *.txt
connect-embed-dist-poly.svg        connect-embed.svg  dataXdata_380.html   dataXtrain.npy   embed_950.html             embed.html  train2.txt            trainXtrain_950.html
connect-embed-dist.svg             data2.txt          dataXdata_950.html   dataXtrain.png   embed_data.ann             embed.npy   train.ann             trainXtrain.npy
connect-embed-path-dataXdata.svg   data.ann           dataXdata.npy        embed_1000.html  embed_data.npy             embed.png   train.npy             trainXtrain.png
connect-embed-path-dataXtrain.svg  data.npy           dataXdata.png        embed_350.html   embed_data.pkl             embed.svg   train.pkl             trainXtrain.svg
connect-embed-path.svg             data.pkl           dataXtrain_380.html  embed_380.html   embed_dataXembed_data.npy  pcmp.ann    train.txt
connect-embed.png                  data.txt           dataXtrain_950.html  embed_400.html   embed_dataXembed_data.png  pcmp.pkl    trainXtrain_380.html

Pretty much every one of these files was important at some point, but quickly became unmanageable.

It would be really nice if there was a better way of organizing my input and output files (datasets and images). Namely, I want to avoid dealing with unique filenames.

The same script that generates data.npy is used to create train.npy, another for embed_380.html / embed_950.html (preview v.s. full screen), and so on. Instead of managing this stuff myself (it does take mental energy to come up with names), it would be better if my experiments were managed in a central place. Artifact paths from the perspective of my python scripts could be the same, but become unique when uploaded and accessed on cloud storage, with the local copy disappearing.

Moreover, I want to easily download the images I select for inclusion into these posts, download them to my computer, and upload them to a public location. Until then, the artifacts I create should remain privately stored.

I reached for a tool I’ve been using for several years now: clearml. ClearML is an experiment tracking platform (and more), tailor-fit to the Machine-Learning lifecycle. Since there’s tremendous overlap with the aforementioned needs and what this tool provides, I decided to try setting up a new project there.

For the first time, I decided to try using their public cloud as a free-tier user instead of self-hosting (to save on time).

It took over an hour to figure out how to properly manage artifact storage on my own S3 buckets (which are not hosted on AWS, complicating matters), but once I did so I was able to quickly convert the core parts of my workflow to ClearML “tasks.”

I will later turn these into pipelines, but for now I’m still getting a feel for how to structure things. Today’s primary goal is to be able to remove the need for any of the files I listed above to exist on my local storage. These artifacts should be uploaded to remote storage and accessed via the ClearML Web UI, rather than the hacky way I am currently previewing images and handling state-management.

I especially love that I can authenticate against my bucket and download the file from the Web UI directly.

I would add another figure as such to ensure it is deleted after upload:

# ... figure creation

fig.savefig(f"connect-embed-path-{dataset_name}.svg")

task.upload_artifact(
    f"embed_path_black", f"connect-embed-path-{dataset_name}.svg", wait_on_upload=False, delete_after_upload=True
)

Tasks can have configurable parameters which I define, and keeps track of related information (like linking to the dataset used).

from clearml import Task
from _globals import OUTPUT_URI
# ...
task = Task.init(
    project_name="embed",
    task_name="plot",
    output_uri=OUTPUT_URI,
    auto_connect_frameworks={"matplotlib": False},
)
params = {
    "task_id": "524e0ee0baf44362a938258a3ec89714",
    "alpha": 0.2,
    "ftype": "svg",
    "color": "k",
    "figsize": 10,
    "font_family": "Ubuntu Mono",
}
params = task.connect(params)

If I run things with properly isolated optional dependencies, I also get proper tracking of my Python requirements (as well as uncommitted changes!):

$ uv run --isolated --extra plotly --extra clearml task_plot_embed.py

One outcome of this conversion is that my Makefile has far less utility - but on the upside, I can clean it up a lot and just leave the data downloading parts.

One lesson was to take manual control over matplotlib figures (it’s probably best practice to use unique names too for Figure objects):

# ...
task = Task.init(
    project_name="embed",
    task_name="plot",
    output_uri=OUTPUT_URI,
    auto_connect_frameworks={"matplotlib": False},  # this line is important for me
)

And that report_image=True was helpful for previews of artifacts that I publish in each experiment.

# ... figure creation
L = task.get_logger()
figpath.savefig(f"connect-embed-path-{dataset_name}.svg")

task.upload_artifact(
    f"embed_path_black",
    f"connect-embed-path-{dataset_name}.svg",
    wait_on_upload=False,
    delete_after_upload=True,
)
L.report_matplotlib_figure(
    title="Path",
    series=dataset_name,
    figure=figpath,
    report_image=True,
    report_interactive=False,
)

One reason I’ve stuck with ClearML over the years (mostly for work), is because I have seen its potential for managing my artwork and the experiments associated with the long process of going from an idea to a printed image. Almost all of its utility for my purposes comes from the convenience of the API to run experiments and retrieve the outputs that were generated. It allows me to work somewhat messily and still ensure that I can reproduce prior results.

I checked in on the usage from this evening of setting up my initial workflows.

I made a lot of mistakes along the way, which I cleaned up by “Archiving” them, but I only worked on this for about 3 hours.

What I cannot quite understand is why this resulted in anything being stored on ClearML’s servers (since I used my own storage … though it took a few failed attempts that I ensured I deleted), and 14K API calls for only a couple dozen runs (averaging about 500 calls per run).

In theory, this should be OK under the free tier, but I can always revert to self-hosting as I’ve done before. I really am curious to see how much value can be derived out of frequent usage of the cloud-hosted product, and if the metrics/artifacts storage will be enough (especially given that I’m managing my own assets in S3).
