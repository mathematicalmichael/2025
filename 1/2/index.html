Kosmos-2, Part 2

2025-01-02

I cleaned up my process for generating videos using Kosmos-2, and began trying a new video (mostly to give you something else to look at).

To scale my experiments, I need to be able to process videos quickly. This means I need to distribute work across multiple machines (GPUs), but I only have a single one in my server. An option available to me is using cloud GPUs, so I can provision many copies and process frames in parallel, load-balancing the requests. One way to accomplish this is to wrap the functionality of Kosmos-2 in an API, so that I can send it frames and get back descriptions (small JSON payloads, see below).

$ curl -X POST -F "content=@snowman.png" http://127.0.0.1:8020/predict
{
  "generated_text": "<grounding> Describe this image:<phrase> A snowman</phrase><object><patch_index_0044><patch_index_0863></object> is sitting by<phrase> a campfire</phrase><object><patch_index_0005><patch_index_1007></object> in the snow.",
  "entities": [["A snowman",[21,30],[[0.390625,0.046875,0.984375,0.828125]]],["a campfire",[45,55],[[0.171875,0.015625,0.484375,0.984375]]]],
  "output": "Describe this image: A snowman is sitting by a campfire in the snow."
}

By offloading the work to an API in the cloud, in theory I should be able to achieve processing frames in real-time (with some latency).

So to start this effort, I used the python package litserve to create an http endpoint through which to call the model, and began testing things locally on my own hardware to work out various settings. I’ll write up that undertaking in its own post, but you’re probably expecting something to look at by this point…

Results

Allow me to introduce you to the late Betty, a jumping spider I found crawling in my art studio some time in Spring 2024:

I decided to start out with what I’m calling the “standard prompt,” asking Kosmos-2 to describe the scene in detail.

This was an interesting one.

  • The descriptions varied widely in length, even when the scene wasn’t changing much.
  • Some objects mentioned do not correspond to bounding boxes (the answer is not “grounded” in evidence). See Frame 47 below.
  • Temporal stability is really bad. It’s a unique scene, admittedly, but it feels downright chaotic.
  • The “spider” would sometimes be labeled “tarantula,” other times “a small bug,” and yet others not at all.
  • The bounding boxes flickering in and out is a jarring visual.
  • I know the spider is visible in every frame, but detections do drop out. I would be curious to compute the rate.

Let’s do some quick math using the command line to figure out the positive detection rate:

parse:
	@ls out/frames | parallel --jobs 12 jq .output out/frames/{}

$ make parse | grep spider | wc -l
213
$ ls out/frames/ | wc -l
529
$ ec
ho "scale=2; $(make parse | grep spider | wc -l) / $(ls out/frames/ | wc -l)" | bc
0.40

It looks like Kosmos-2 is identifying the spider in only 40% of frames, based on its description of the scene.

If I look at the .entities field, it’s only 23% of the frames that contain a bounding box identified with the word “spider.” I wonder if this would change if I asked it to “find the spider” instead of “describe the scene in detail.”

Let’s save this command for later:

term=spider
ratio:
	@echo "scale=2; $$(make parse | grep -c $(term)) / $$(ls out/frames/ | wc -l)" | bc

And re-run the pipeline to see if the spider is detected more frequently:

I switched .output for .entities in the parse command, since now “Spider” was in the prompt. The number to beat is 0.23 (against the phrase “Describe the scene in detail”).

make ratio term=spider
0.37

37% is a substantial improvement over 23%, but it’s still not great.

Of course this doesn’t count phrases like “bug” or “tarantula” but the point remains: this model is compelling in that it allows for flexible prompts, but it’s certainly not magic with a simple “describe the scene in detail” prompt. I am curious if a longer prompt (regarding what kinds of descriptions it should give), would change the generic results.

The output of this prompt was odd: the annotation often seemed strongly conditioned by the “Macbook Pro” (it was an Air) on the zoomed-out parts of the video.

Concluding Thoughts

I suppose I’m circling around this question:

Can I engineer a prompt that works well for some artistic use-case as a generic scene descriptor?

I may have to consider the potential use-cases of this model carefully, and need to test it on a wider variety of images and videos. The idea of real-time video processing using this algorithm is compelling, but I do hold aesthetics in high regard and do not find the video’s instability visually compelling. I spent a bit more time refining the tooling to show what you see here than actually playing around, so you can expect more results soon.
